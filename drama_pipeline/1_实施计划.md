## Recommendation Algorithm Revision

This implementation was later revised by `1_鎺ㄨ崘璇勫垎璋冩暣璁″垝.md` to fit the current business goal of absolute revenue maximization.

Current recommendation strategy:

- Candidate sources are split by responsibility:
  - `Mobo/鍖楁枟鏂板墽`: newness and age buckets.
  - `澶氫箰鎸夊ぉ姒渀銆乣澶氫箰 DramaBox 姒渀: first capture `1000` raw rows per sheet, then keep `600 / 200 / 400` by sheet for date-based explosion evidence.
  - `椋炰功鐑棬閾炬帴`: parse by day blocks; only rows whose first column is a date count as recommendation rows, header rows start with `鎺ㄨ崘鏃ユ湡`, and note rows are ignored.
  - `椋炰功鍖楁枟鑻辫鐑`: English-only Beidou hot evidence.
  - `Mobo 鎺ㄨ崘`: current recommendation strength (`Top 1~50 / 51~100 / 101~500`).
  - `鍖楁枟鏀跺叆姒渀: non-English only; weak revenue validation plus explosion assist (`<=200 涓?<=28澶ー銆乣<=400 涓?<=60澶ー銆乣<1000`). Together with the Feishu English hot sheet it forms the full `鍖楁枟鐖嗘` pool.
- Age buckets stay fixed at `1~14` / `15~28` / `29+`.
- Candidates are first mapped into A~G layers:
  - `A 鏂?鏂扮垎`
  - `B 涓?鏂扮垎`
  - `C 鏂?鍏朵粬`
  - `D 涓?涓垎`
  - `E 涓?鍏朵粬`
  - `F 鑰佸墽鐖嗘鍓╀綑姹燻
  - `G 鏅€氬墿浣欏墽`
- Layer order is fixed: `A > B > C > D > E > F > G`.
- A~E layer scoring:

```text
layer_score = 45% explosion_recommend + 35% freshness + 20% revenue_validation
final_score = layer_score * theme_multiplier * content_multiplier
```

- F layer uses stronger old-hit weighting:

```text
layer_score_F = 55% explosion_recommend + 15% freshness + 30% revenue_validation
final_score_F = layer_score_F * theme_multiplier * content_multiplier
```

- G layer is a pure fallback ordered by newest promotion time first, then theme/content multipliers, then weak recommendation hints.
- Each run should also emit debug counts for every fetch and processing stage: raw source rows, parsed rows, post-dedupe rows, post-filter rows, A~G layer counts, material-qualified rows, and final recommendation rows.
- Theme scoring now acts as a multiplier instead of the old `0.5~1.2` tag score:
  - default `1.0`
  - range `0.8~1.2`
  - revenue amount drives preference direction
  - order count only affects confidence shrinkage
- Content handling is no longer a hard AI/anime block:
  - adult/explicit titles: hard block
  - AI: multiplier `0.8`
  - anime/comic: multiplier `0.6`
  - both AI and anime: multiplier `0.6`
- Final selection follows the current `LANGUAGE_THEATER_QUOTAS` allocation across 7 languages, and the active total target is `66` titles.
- Final selection still respects the existing per-language theater quotas from `LANGUAGE_THEATER_QUOTAS`, without adding a separate global `Moboreels <= 4` cap.

Current implementation override on 2026-04-22:

- The effective final target is no longer the old `70`-title placeholder. The run now follows `LANGUAGE_THEATER_QUOTAS` directly, and the current total target is `66`.
- Material qualification now defaults to full pass without requesting the material endpoint. Candidates that pass the other filters continue as qualified, and the workflow records the run as material-skipped.
