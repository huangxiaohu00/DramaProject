## 一、项目概览

### 1.1 项目定位
短剧推荐与订单追踪流水线系统，服务于多个海外短剧平台（MoboReels、DramaBox、ShortMax、FlareFlow、ReelShort、HoneyReels、SnackShort、StarShort），覆盖7种语言、8个剧场。

### 1.2 核心业务流程
| 流程 | 功能 | 入口 |
|------|------|------|
| 今日推荐 (Today Recommendation) | 从4个数据源获取候选剧→去重→过滤→评分→分层→配额→素材验证→输出推荐清单 | `6_today_recommend.py` |
| 昨日订单 (Yesterday Orders) | 从Mobo和北斗多账户获取订单→聚合→输出Excel汇总 | `7_yesterday_orders.py` |

## 二、项目规划评估

### 2.1 优点

- **规划文档完善**：三份规划文档（项目规划、实施计划、评分调整计划）覆盖了目标、架构、任务分解和算法演进，这在同规模项目中少见。
- **TDD驱动的实施流程**：实施计划严格遵循"先写失败测试→验证失败→实现→验证通过"的TDD纪律，确保代码可测。
- **模块职责清晰**：数字前缀命名（2_~9_）使文件在资源管理器中按逻辑顺序排列，便于业务人员浏览。
- **渐进式迁移**：15步迁移计划允许新旧系统并存，降低切换风险。
- **风险识别到位**：规划文档明确识别了Token过期、Duole网页抓取不稳定、字段变更等5类风险。

### 2.2 问题

| 级别 | 问题 | 说明 |
|------|------|------|
| **严重** | **无版本控制** | 项目明确标注不是git仓库，所有修改直接写入磁盘，无法回溯、无法diff、无审计追踪 |
| **高** | **数字前缀文件名** | `2_config.py` 等名称导致无法使用标准Python import，被迫使用 `importlib.import_module()`，破坏IDE自动补全、静态分析、依赖追踪 |
| **高** | **缺乏部署/运维规划** | 无CI/CD、无容器化、无定时任务调度方案（需人工触发CLI） |
| **中** | **文档与实现不同步** | `basic_data/drama_recommendation_scoring.md` 描述的是旧评分体系（0.5兜底、0~1权重），实际已切换为0.8~1.2系数体系 |
| **中** | **缺少错误恢复计划** | Token过期、Cookie失效等运行时故障无自动恢复机制，仅靠人工感知和手动更新 |

### 2.3 评分：7.5/10

规划文档的完整性和TDD方法论是亮点，但无版本控制和数字文件名是根本性架构缺陷。

## 三、项目逻辑评估

### 3.1 今日推荐流程

```
数据采集(4源并发) → 去重(语言+标题+剧场) → 过滤(已发布/成人内容)
→ 评分(爆炸+新鲜度+收入验证) → 分层(A~G) → 配额检查 → 素材验证 → 输出
```

**优点**：
- 并发采集使用 `ThreadPoolExecutor`，4个平台并行拉取，效率高
- 分层机制（A~G）逻辑清晰，新剧高分剧优先，老剧需额外爆炸信号支撑
- 配额系统支持语言级+剧场级双重限制，避免某语言/剧场过度推荐
- 素材验证环节可独立开关（`MATERIAL_CHECK_ENABLED`），默认配置下视为全部达标且不请求素材接口

**问题**：

| 级别 | 问题 | 影响 |
|------|------|------|
| **中** | 语言配额达到总量后拒绝该语言所有候选，可能导致某些剧场配额永远填不满 | 推荐覆盖不均 |
| **中** | `material_cache` 在推荐循环中被惰性填充，是迭代中的副作用 | 逻辑不透明 |
| **低** | `build_default_clients()` 直接取 `config.MOBO_AUTHS[0]` 无空列表保护 | 潜在IndexError |
| **低** | `_resolve_material_counts` 返回4元组，可读性差 | 维护困难 |

### 3.2 昨日订单流程

```
多账户并发拉取 → 聚合(日期+平台+语言+剧场+标题) → 求和+合并账户 → 输出
```

**优点**：
- 支持多Mobo+多北斗账户并发拉取，充分利用多凭证
- 聚合逻辑使用 `OrderedDict` 保持插入顺序
- 错误隔离：单个账户失败不影响其他账户

**问题**：

| 级别 | 问题 | 影响 |
|------|------|------|
| **中** | `client.account = f"mobo_{index}"` 是动态属性，未在类中声明 | 脆弱，依赖Python动态特性 |
| **低** | `export_orders` 函数未被调用，是死代码 | 代码冗余 |
| **低** | 无并发限速，9个账户同时请求可能触发平台限流 | 运行时风险 |

### 3.3 评分：7/10

业务流程清晰完整，并发设计合理，但存在配额逻辑缺陷和代码副作用问题。

## 四、算法优劣评估

### 4.1 推荐评分算法（核心）

#### A~G分层体系

| 层 | 含义 | 条件 |
|----|------|------|
| A | Fresh Challenger | 新剧+多源爆炸 |
| B | Fresh Star | 新剧+单源爆炸 |
| C | Rising Challenger | 中期剧+多源爆炸 |
| D | Rising Star | 中期剧+单源爆炸 |
| E | Established | 中期剧+无爆炸但高评分 |
| F | Old Winner | 老剧+强爆炸信号 |
| G | Fallback | 老剧/无爆炸，按推广时间倒序 |

#### 层内评分公式
- **A~E层**: `layer_score = 45% × 爆炸推荐分 + 35% × 新鲜度分 + 20% × 收入验证分`
- **F层**: `layer_score = 55% × 爆炸推荐分 + 15% × 新鲜度分 + 30% × 收入验证分`
- **最终分**: `final_score = layer_score × 主题系数 × 内容系数`

#### 爆炸信号合成
`爆炸分 = 60% × 日期信号 + 25% × Mobo推荐位 + 15% × 北斗验证`

**优点**：
- 分层机制确保推荐优先级符合业务直觉（新剧优先、爆炸信号加持）
- F层对老剧加大爆炸权重（55%）是合理的——老剧需要更强的市场信号
- 主题系数（0.8~1.2）和内容系数（AI=0.8, 动漫=0.6）提供了精细化调控
- 排名评分使用对数衰减 `1 - (1-min) × log1p(rank-1) / log1p(max-1)`，避免线性衰减过于陡峭

**问题**：

| 级别 | 问题 | 说明 |
|------|------|------|
| **高** | 爆炸信号权重(0.6/0.25/0.15)硬编码在规则模块而非配置 | 修改需改代码，非配置驱动 |
| **中** | 日期信号只检查北斗热门和Duole推荐日期，未考虑Mobo推荐日期 | 信号源不全 |
| **中** | `_content_text` 将raw dict所有值拼接后做关键词匹配，可能在URL/ID等字段误匹配 | 如URL中含有"ai" |
| **中** | `score_beidou_validation` 对不同age_bucket使用不同阈值，但阈值硬编码 | 缺乏可配置性 |
| **低** | `dedupe_candidates` 合并raw dict时用 `raw.update()`，同key不同值会静默丢失 | 数据完整性 |

### 4.2 标签评分算法

**核心逻辑**：
1. 标签归一化 → 四级证据查找（语言+剧场→语言→剧场→全局）
2. 每级：`校准分 = 全局分 + 级别置信度 × 样本置信度 × (级分 - 全局分)`
3. 多标签：Top-3标签加权衰减 `(1.0, 0.5, 0.25)`
4. 钳制范围 `[0.8, 1.2]`

**优点**：
- 四级证据体系优雅地解决了冷启动问题（数据少的标签自动回归全局均值）
- 贝叶斯风格的校准公式有统计学依据
- 多标签衰减加权避免被大量低质量标签稀释

**问题**：

| 级别 | 问题 | 说明 |
|------|------|------|
| **中** | TagWeightScorer每次实例化都重新读取Excel文件 | 性能浪费，`score_candidates`默认每次新建实例 |
| **中** | workbook不存在时静默返回默认值，无警告日志 | 故障不可感知 |
| **低** | 全局权重构造中 `0.85 × 金额 + 0.15 × 订单数` 的权重硬编码 | 应可配置 |

### 4.3 算法整体评分：8/10

分层评分体系设计精良，统计学基础扎实。主要缺陷是关键参数硬编码和部分信号源遗漏。

## 五、代码规范评估

### 5.1 优点

- **类型注解完整**：所有函数签名都有参数和返回值类型注解
- **数据类设计规范**：正确使用 `field(default_factory=...)` 避免可变默认值陷阱
- **函数式倾向**：规则模块（8_recommendation_rules）使用纯函数，易于测试
- **统计追踪**：FetchStatsMixin统一记录各源拉取数据量，利于运维
- **CLI入口**：每个编排模块都有 argparse + main()，支持命令行独立运行
- **测试断言具体**：测试验证精确数值而非"不为空"，回归保护能力强

### 5.2 问题

#### 5.2.1 安全问题（严重）

| 问题 | 位置 | 说明 |
|------|------|------|
| JWT Bearer Token硬编码 | `2_config.py:62-63, 70-71` | 完整JWT，包含用户ID和角色信息 |
| Feishu app_id/app_secret硬编码 | `2_config.py:78-79` | 可生成租户访问令牌 |
| Duole Cookie硬编码 | `2_config.py:96` | 完整会话Cookie含CSRF令牌 |
| Material API Cookie硬编码 | `2_config.py:149` | 仅此一个有环境变量覆盖 |
| **所有凭证应迁移至环境变量或密钥管理器** | - | 当前任何获得代码访问权的人即拥有全部API权限 |

#### 5.2.2 架构问题

| 问题 | 影响 |
|------|------|
| `importlib.import_module()` 替代标准import | 破坏IDE自动补全、静态分析、依赖图生成、重构工具 |
| 无包结构（缺少 `__init__.py`） | 无法作为Python包安装和引用 |
| `DramaRecord` 24个字段的"上帝对象" | 数据模型耦合度高，生命周期跨越整个流水线 |
| JavaScript内嵌在Python代码中（~200行） | 不可维护、不可测试、IDE不可见 |

#### 5.2.3 错误处理

| 问题 | 位置 | 说明 |
|------|------|------|
| `except Exception` 过宽 | `4_platform_clients.py` 的 `_safe_int/_safe_float` | 应仅捕获 `TypeError, ValueError` |
| 无重试机制 | 所有API调用 | 瞬态网络故障直接失败 |
| 无限速 | 并发API调用 | 可能触发平台限流 |
| 无日志系统 | 全局 | 仅用 `print()`，无法控制级别、重定向、生产环境调试 |

#### 5.2.4 代码细节

| 问题 | 位置 | 说明 |
|------|------|------|
| `__import__("datetime")` 替代顶部import | `4_platform_clients.py:1466`, `6_today_recommend.py:260` | 违反PEP8，降低可读性 |
| `THEATER_NAMES` 中 `"sanckshort"` 拼写错误 | `2_config.py` | 虽可作为别名工作，但令人困惑 |
| `THEATER_NAME_TO_ID` 键碰撞 | `2_config.py` | 同一display name对应多个slug，后者覆盖前者 |
| 剧场名模糊匹配可能误匹配 | `2_config.py:280` | `alias in compact or compact in alias` 子串匹配有误报风险 |
| 飞书仅保留英语热门 | `4_platform_clients.py:553` | `language != "英语"` 过滤丢弃所有非英语热门剧 |
| 死代码 `export_orders` | `7_yesterday_orders.py:168-171` | 从未被调用 |
| `_explosion_priority` 每次调用重建字典 | `8_recommendation_rules.py:582-584` | 应为模块常量 |
| 测试中使用 `importlib.import_module` | 所有测试文件 | 同样的工具链问题 |

#### 5.2.5 线程安全

`FetchStatsMixin.fetch_stats` 列表在 `ThreadPoolExecutor` 多线程中被修改，无加锁保护。依赖CPython GIL的列表append原子性是实现细节而非语言保证。

### 5.3 代码规范评分：5/10

类型注解和数据类设计良好，但安全问题严重、import机制反模式、无日志系统是重大扣分项。

## 九、结论

Drama Pipeline 是一个**业务逻辑完整、算法设计精良**的数据流水线项目。A~G分层推荐体系和贝叶斯风格的标签评分算法展现了扎实的业务理解和统计学功底。TDD驱动的开发流程和精确的数值断言测试保证了核心逻辑的正确性。

然而，项目在**工程化维度存在明显短板**：凭证硬编码是安全红线，数字前缀文件名破坏了Python生态工具链，无版本控制是最大的运维风险，缺乏日志系统使得生产调试几乎不可能。

**核心矛盾**：这是一个"业务正确性优先"的项目——算法和流程经过反复验证，但工程基础设施尚未跟上业务成熟度。优先级应放在安全（凭证外迁）和版本控制上，然后逐步补齐工程化短板。
